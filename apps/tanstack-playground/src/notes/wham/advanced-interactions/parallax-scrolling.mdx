## TL;DR

- Parallax = 不同「深度」的圖層用不同速度位移：遠的（天空）幾乎不動，近的（前景樹）位移大。視差差異欺騙視覺系統產生「景深」錯覺。
- Pre-`animation-timeline` 寫法：JS 監聽 `scroll`，每幀讀 `scrollY` 算 `translate` 套到每層 → main thread bottleneck、容易丟幀。`animation-timeline: scroll()` 把整套外包給 CSS engine，瀏覽器拿 compositor thread 跑，smooth 到底。
- 用一份 `@keyframes parallax { from { transform: translateY(var(--from)); } }` 配 inline style `style={{ '--from': '50px' }}` — 每層讀自己的 CSS custom property，就地產生不同程度的位移。
- `--from` 值的直覺：scroll 到底時所有層都歸位 `translateY(0)`，所以 `--from` 是「scroll 開始時這層比 resting 位置多偏移多少」。值大 = 移動範圍大 = 視覺上「近」；值小 = 幾乎不動 = 視覺上「遠」。
- **`prefers-reduced-motion` 不是 polish 是必要**。Parallax 在 vestibular sensitivity（前庭敏感、暈眩）的觸發名單上排第一，比一般 motion 嚴重很多。要 wrap 在 `@media (prefers-reduced-motion: no-preference)` 後面，不然 reduced-motion 使用者直接被晃吐。

## 為什麼以前做這個會頭痛

Pre-`animation-timeline` 時代的 parallax recipe：

```ts
window.addEventListener('scroll', () => {
  const y = window.scrollY;
  skyLayer.style.transform = `translateY(${y * 0.05}px)`;
  mountainLayer.style.transform = `translateY(${y * 0.3}px)`;
  treesLayer.style.transform = `translateY(${y * 0.7}px)`;
});
```

問題：

1. `scroll` event 在 trackpad / touchscreen 觸發頻率非常高（一秒 100+ 次），每次都跑 JS callback + 寫 style。
2. 寫 `transform` 強制 layout / paint / composite — 雖然 transform 通常是 compositor-only，但 main thread 排程一爆 frame 就掉。
3. 結果是「scroll 看起來會卡卡的」— 經典的 jank。一張靜態圖看起來丟了 3–4 幀，立刻被察覺。

解法歷代都不漂亮：throttle scroll handler（變不順）、`requestAnimationFrame` 包一層（緩解但沒治本）、用 IntersectionObserver 配 CSS transition（限制多）。

`animation-timeline: scroll()` 直接把整個機制下放到 CSS engine — 動畫進度跟 scroll progress 綁定，瀏覽器拿 compositor thread 跑，main thread 幾乎不動。

## 一份 keyframes，多層各自的 `--from`

關鍵 trick 是 CSS custom property 可以塞進 keyframes：

```css
.scroller {
  overflow-y: scroll;
  scroll-timeline-name: --parallax-scroll;
  scroll-timeline-axis: block;
}

@keyframes parallax {
  from { transform: translateY(var(--from)); }
  to   { transform: translateY(0); }
}

.layer {
  position: absolute;
  inset: 0;
  animation: parallax linear;
  animation-timeline: --parallax-scroll;
}
```

兩個踩過的雷：

1. **匿名 `animation-timeline: scroll()` 在 sticky 父容器內解析失敗**。`scroll()` 預設綁「nearest scrollable ancestor」，但實測 layer 在 `position: sticky` 的 stage 內時 Chrome 解不到正確的 scroll source — `getAnimations()[0].currentTime` 永遠是 `null`、layer 完全不動。改用 **named timeline**（`scroll-timeline-name: --parallax-scroll` 在 scroller 上 + `animation-timeline: --parallax-scroll` 在 layer 上）明確綁定就修好。
2. **`to` 不要省**。理論上空 `to` 該 fallback 到 element baseline (`translateY(0)`)，但 Chrome 對「keyframe 用 `var()` + scroll-driven animation」這個組合的 fallback 似乎不穩。寫死 `to { transform: translateY(0); }` 比較踏實。

每層只要在 inline style 給不同的 `--from`：

```tsx
<div className="layer" style={{ '--from': '0px' }}>   {/* sky */}
<div className="layer" style={{ '--from': '10px' }}>  {/* far clouds */}
<div className="layer" style={{ '--from': '16px' }}>  {/* near clouds */}
<div className="layer" style={{ '--from': '50px' }}>  {/* far mountains */}
<div className="layer" style={{ '--from': '70px' }}>  {/* mid mountains */}
<div className="layer" style={{ '--from': '130px' }}> {/* foreground */}
```

CSS engine 看到 keyframe 內的 `var(--from)`，會去讀**每個 element 自己的** `--from` 值。一份 keyframe 定義服務六層，每層位移不同。

TS 上 `style={{ '--from': '50px' }}` 會被 React 的 type 拒絕，因為 `--from` 不在 `CSSProperties`。我這個 demo 用 `style={{ '--from': '50px' } as LayerStyle}` 把它 cast 成擴充過的 type；也可以用 `as React.CSSProperties` 強制蓋掉，看你 codebase 的偏好。

## `--from` 值怎麼挑

Scroll 到底時所有層都到 `translateY(0)`（也就是 keyframes 的 `to` / element baseline）。所以 `--from` 是「scroll 起點時這層比 resting 多偏移幾 px」。

直覺映射：

- 遠的東西在現實中視差小（看遠方雲，車跑了 10 公尺位置幾乎沒變）→ `--from` 小
- 近的東西視差大（看路邊樹，車跑了 10 公尺整個刷過去）→ `--from` 大

我這個 demo 用 `0 / 10 / 16 / 50 / 70 / 130`，從天空到前景樹差近 130 倍。實際數字是 vibes-based — 拉到不舒服就改小、看不出視差就改大。Comeau 課程也是直接給數字，沒有公式。

## sticky 容器讓圖層留在畫面上

`animation-timeline: scroll()` 預設綁定**最近的可捲動祖先**的 scroll 進度。但圖層本身要怎麼留在使用者視線範圍內？兩個寫法：

1. `position: fixed`：圖層脫離 layout 釘在 viewport，scroll 時圖層不會被捲走。**問題**：fixed 是相對 viewport，圖層會蓋掉整個頁面，不是只蓋 demo 容器。
2. `position: sticky` + 有 `overflow` 的祖先：圖層會在祖先範圍內被「黏住」上緣，捲到祖先邊界外就跟著走。我這個 demo 走 sticky 路線。

我的 DOM 結構：

```html
<div class="parallax-scroller">          <!-- overflow:scroll, height:320 -->
  <div class="parallax-track">           <!-- height:1200, 提供 scroll range -->
    <div class="parallax-stage">         <!-- position:sticky, top:0, height:320 -->
      <div class="parallax-layer">...</div>   <!-- 6 個 absolute 圖層 -->
    </div>
  </div>
</div>
```

`.parallax-track` 是 1200px 高的「捲動跑道」、`.parallax-stage` 用 sticky 在跑道上端釘住可視區、6 個 layer 是 stage 的 absolute 子元素。User 捲動時 stage 釘在頂部 → 看起來整個風景卡在畫面上、layer 各自 transform 產生視差。

## browser support 怎麼降級

`animation-timeline` 在 2024–2025 才陸續普及。Chrome / Edge 已支援，Firefox / Safari 還在路上（2026 上半年觀察 Firefox 已 ship、Safari 有 flag）。沒支援的瀏覽器拿到 `animation-timeline: scroll()` 會直接忽略 — 結果是 layer **全部停在 baseline**（`translateY(0)`），看起來是一張正常風景圖、沒有視差但也沒壞。Graceful 降級不需要寫額外 fallback。

不過要注意 `var(--from)` 在 keyframe 內部目前 polyfill 並不友善 — 如果想 polyfill 給舊瀏覽器，CSS custom property 在 keyframes 的取值時機通常拿不到，會變每層 `translateY(0)` 沒視差，跟 baseline 一樣。換句話說：要嘛裝最新瀏覽器吃原生支援，要嘛就接受是平面圖 — 別嘗試 polyfill 這套。

## prefers-reduced-motion 是不能省的

Comeau 在課裡明說這不是 grey area：parallax 是 **vestibular motion sickness 的頭號觸發**，比一般 transition 嚴重很多。前庭敏感的人遇到 parallax 不是「覺得有點晃」，是會立刻頭暈反胃。一定要 wrap：

```css
@media (prefers-reduced-motion: no-preference) {
  .parallax-layer {
    animation: parallax linear;
    animation-timeline: scroll();
  }
}
```

注意 query 寫的是 `no-preference` 不是 `reduce` — 這樣預設是「不開」、只有 user 明確沒設定 reduced-motion 才開。比 `@media (prefers-reduced-motion: reduce) { animation: none; }` 安全：那種寫法如果 user 還沒選擇偏好（某些舊瀏覽器、某些 OS），會 fall through 到開啟 — 違反 a11y safe-by-default 原則。

## Gotchas

- **匿名 `scroll()` 配 sticky 父容器會啞掉**。layer 在 `position: sticky` 的 stage 裡時，`animation-timeline: scroll()` 預設的「nearest scrollable ancestor」解析機制在 Chrome 上拿不到實際 scroll source — `getAnimations()[0].currentTime` 永遠 null、layer 完全不動。修法：scroller 上 `scroll-timeline-name: --foo`、layer 上 `animation-timeline: --foo`，明確綁定。Debug 訊號：在 console `el.getAnimations()[0].currentTime` 看是不是 null，是的話就是這個雷。
- **keyframe `to` 不要靠 implicit fallback**。理論上空 `to` 該 fallback 到 element baseline，但 Chrome 對「`var()` + scroll-driven animation」這個組合的 fallback 不穩，computed transform 直接 `none`。寫死 `to { transform: translateY(0); }` 一勞永逸。
- **`prefers-reduced-motion: no-preference` 是 mandatory, 不是 nice-to-have**。Parallax 在前庭敏感觸發名單排第一，課程 explicit 拉出來講「這不是 grey area」。Wrap 整段 `animation` + `animation-timeline` 在 `no-preference` query 後面，預設關，user 明確 opt-in 才開。
- **CSS custom property 在 keyframes 的 polyfill 不可靠**。`@keyframes` 用 `var(--from)` 大部分 polyfill（包括 Bramus 的 `scroll-timeline.js`）拿不到 element-scope 的值、退化成 0。降級路徑就是平面圖，別硬 polyfill。
- **`position: sticky` 受 overflow scope 限制**。Sticky 黏的是「最近有 scroll 機制的祖先」，跟 `position: fixed` 不一樣（fixed 直接黏 viewport）。如果 demo 容器外有自己的 transform / opacity / filter，會 break sticky context — sticky 元素會直接變 static。
- **Layer 順序 = DOM 順序 = z-stacking 順序**。最後一個 layer 蓋住前面，所以 DOM 內要從遠（天空）到近（前景樹）依序排，`absolute` + `inset: 0` 自然堆疊。不要靠 `z-index` 排，layer 數一多 z-index 失控。
- **`--from` 數字憑感覺挑、沒公式**。我這個 demo 用 `0 / 10 / 16 / 50 / 70 / 130` 跨 6 層；課程示範也是手工 tune。理論上可以用 `1 / depth` 之類 formula 但實務沒人這樣寫 — 直接看畫面調最快。
- **Horizontal parallax 也成立**。`@keyframes` 改 `translateX(var(--from))` 即可，配 `animation-timeline: scroll(inline)` 抓水平 scroll。Web 上少見（垂直 scroll 是預設 mental model），但遊戲 side-scroller 概念一樣。
- **`image-rendering: pixelated` 只對 raster 有用**。如果用 PNG / pixel-art sprites 想保持 pixelated 銳利，加這個避免 GPU 縮放糊掉；但 SVG 是向量，scale 不需要這個 hint，加了沒影響。
