## TL;DR

- `<canvas>` 有兩套尺寸:CSS 尺寸 vs `width`/`height` attribute (內部解析度)。後者只在 `setupCanvas` 那一刻算一次，window 變大後就被 GPU 拉伸糊掉。
- 解法是把 setup 邏輯包成可重跑的 `update()`，掛 `window.addEventListener('resize', update)`。setupCanvas 額外回傳 `disposeResize` 給 React `useEffect` cleanup。
- 邊界判斷如果只翻 velocity 不 clamp 位置，window 縮到 cut off 元素的瞬間 velocity 會 ping-pong 卡邊;補一句 `logo.x = canvasDimensions.width - logo.size` 推回畫布內。

## Canvas 為什麼一 resize 就糊

`<canvas>` 有兩套尺寸:

1. **CSS 尺寸** (`w-full h-[240px]` 之類的 style) — 螢幕上看到多大
2. **內部解析度** (`canvas.width` / `canvas.height` attribute) — 真正畫了多少 pixel

mount 時 `setupCanvas` 算一次，內部解析度被釘在當下的 CSS 尺寸 × DPR。Window 放大後，CSS 尺寸跟著大，內部解析度沒變;瀏覽器把那批 pixel 拉伸鋪到新空間，看起來就糊。

修法是把 setup 拆成可重跑的 `update`:

```ts
function update() {
  const dpr = window.devicePixelRatio || 1;
  const rect = canvas.getBoundingClientRect();
  canvas.width = Math.round(rect.width * dpr);
  canvas.height = Math.round(rect.height * dpr);
  ctx.scale(dpr, dpr);
  dimensions.width = rect.width;
  dimensions.height = rect.height;
}

window.addEventListener('resize', update);
update();
```

`setupCanvas` 多回傳一個 `disposeResize`，是 `removeEventListener` 的 wrapper，給 `useEffect` cleanup 用 — 不然 Strict Mode double-mount + unmount 會留 listener。

## 為什麼 dimensions 一定要是物件

`setupCanvas` 只 return 一次，但 width/height 在 resize 之後要持續更新。

```ts
// ❌ primitive 不行
return { ctx, canvasWidth, canvasHeight };
```

JS primitive (number) 是 immutable。`canvasWidth = newWidth` 只是把這個 local 變數重新綁到一個新數字，consumer 已經拿到手的那個 snapshot 不會變。

```ts
// ✅ 物件 by reference
const dimensions = { width: 0, height: 0 };
function update() {
  dimensions.width = rect.width;
  dimensions.height = rect.height;
}
return { ctx, dimensions };
```

物件是 reference type，`update` 改的是那塊記憶體本身，consumer 透過 `canvasDimensions.width` 每次讀都拿到最新值。也是為什麼 demo 的 `ctx.clearRect(0, 0, canvasDimensions.width, canvasDimensions.height)` 不用知道有 resize 也照樣對。

## Boundary 還會被打到一個坑

直觀的寫法:撞到邊界就翻 velocity。

```ts
if (logo.x < 0 || logo.x > canvasDimensions.width - logo.size) {
  logo.xVelocity *= -1;
}
```

平常 OK，但 window 縮小到「cut off 已經在飛的元素」的瞬間，`logo.x > canvasDimensions.width - logo.size` 一直是 true，velocity 翻來翻去都還在外面，看起來像卡在邊邊抖。

修法是除了翻 velocity，也把位置 clamp 回畫布內:

```ts
const isBeyondLeft = logo.x < 0;
const isBeyondRight = logo.x > canvasDimensions.width - logo.size;
if (isBeyondLeft || isBeyondRight) {
  logo.xVelocity *= -1;
  logo.x = isBeyondLeft ? 0 : canvasDimensions.width - logo.size;
}
```

下一幀位置貼在牆上 (不再越界)，條件不會再觸發，velocity 翻完就正常往回飛。

## Gotchas

- **`canvas.width = ...` 會清空 canvas 全部狀態** — 包含 `ctx.scale`、currentTransform、currentPath。這是 spec 行為，所以 `update()` 裡每次都要 `ctx.scale(dpr, dpr)`，不能省。
- **resize listener 一定要 cleanup**。setupCanvas 改回傳 `disposeResize`，React `useEffect` cleanup 一律呼叫，不然 listener 累積。
- **`ResizeObserver` 看起來更對但會閃屏** — 跟 paint timing 不同步，resize 中間會看到一閃。`window` 的 `resize` event 沒這問題。
- **emoji glyph 不一定剛好 = `font-size`**。`ctx.fillText` 渲染的 emoji 跟 `font-size` 是「請給我這麼大」，實際 bounding box 會略小。boundary 用 `LOGO_SIZE` 當寬度近似就好，不用糾結 1-2 px。
- **`canvas.getContext('2d')` 回傳 `null` 在 closure 內無法 narrow** — TS 的 control flow analysis 不會穿透到 closure，所以 `update()` 裡只能用 `ctx!` 或 hoist 一個 narrowed local。
