## TL;DR

- CSS transition 動不了「property 切換」型的 layout 變化（例如 `position: static` ↔ `absolute`、`width: 120px` ↔ `width: revert`）— 沒中間值可以 tween。
- `<motion.div layout>` 直接幫你做。背後是 layout projection（FLIP 的 successor）：量切換前 bounding box、量切換後、用 CSS transform 從舊 box tween 到新 box。看起來像 width / height 在動，實際是 transform。
- transform 把子內容（包含文字）當「螢幕截圖」拉伸 → 文字會 warp。修法：內層放一個 `<motion.p layout="position">`，它套上「反向 transform」抵消父層 scale。
- `transition` prop 不會 inherit。父子都要寫一份相同的 `SPRING`，否則父子用不同 spring，動畫看起來不同步。

## CSS transition 為什麼動不了

最直觀的寫法是 `.maximized { position: absolute; inset: 0; width: revert; height: revert; }`，加 `transition: all 300ms`。實際試會發現：完全沒動，瞬間切換。

原因是 transition 需要「同個 property 從 A 值 tween 到 B 值」。`position: static → position: absolute` 中間沒有有效值;`width: 120px → width: revert` 也沒有。所以瀏覽器直接用切換後的值 paint，沒有過渡幀。

要繞過去，得**改個物件的 transform**而不是改 layout property。但 transform 不影響 layout — 怎麼讓它看起來像「box 變大」?

這是 FLIP 的核心觀察：先量舊 box、套上新 class（瞬間切換 layout）、再量新 box，然後**用 transform 把元素「拉回」舊 box 的位置和尺寸**，最後 tween transform 到 identity。視覺上像 box 從舊狀態漸變到新狀態，但 layout 其實在第一幀就已經切完了。

## `layout` prop 的核心機制

```tsx
<motion.div layout transition={SPRING} className={isMaximized ? 'maximized' : ''}>
  ...
</motion.div>
```

每次 React render 後 motion 比較 bounding box，發現變了就：

1. 量舊位置 / 尺寸（render 前 cached）
2. 套用新 className（DOM 已是新 layout）
3. 量新位置 / 尺寸
4. 設 `transform: translate(deltaX, deltaY) scale(deltaW, deltaH)` — 把元素視覺拉回舊 box
5. 在下一幀把 transform tween 到 `none`

整個過程 layout property 只在第 2 步切一次，剩下都是 transform。Compositor-only animation = 60fps + GPU。

Motion 用的是 layout projection（不是嚴格 FLIP — FLIP 只處理 position 不處理 size），但概念是同一個。

## 子內容為什麼會 warp

父層 `motion.div` 套上 `scale(2, 1)`（變寬 2 倍、高度不變）時，**子內容也跟著被 scale** — `<p>Hello world</p>` 的每個 glyph 都被橫向拉成兩倍寬。瀏覽器不知道這是文字、有 typography 規則、不該被拉伸 — 對它來說 `scale` 是把整個 sub-tree 當一張 raster 處理。

修法是內層再放一個 `motion.p`，告訴 motion「這個子元素的 *position* 你也幫我管，但別繼承父的 *size* transform」：

```tsx
<motion.div layout transition={SPRING} className={...}>
  <motion.p layout="position" transition={SPRING}>
    Hello world
  </motion.p>
</motion.div>
```

`layout="position"` 在 render 後計算自己的「期望位置 vs 父被 scale 後實際位置」，套上**反向 scale + translate** 抵消父的形變。視覺上 p 維持原本字寬，只跟著父的「中心點」移動。

`layout` 三個值差別：

| value | 處理 | 用途 |
|---|---|---|
| `true` | position + size 都動 | 預設，box 整個變形 |
| `"position"` | 只動位置 | 文字 / icon — 不要被 scale 拉變形 |
| `"size"` | 只動尺寸 | 罕見，例如父跟著子尺寸自動撐開 |

## 為什麼父子都要寫 `transition`

```tsx
<motion.div layout transition={SPRING}>
  <motion.p layout="position" transition={SPRING}>{/* ← 這個別漏 */}
    Hello world
  </motion.p>
</motion.div>
```

每個 `motion.*` 元件獨立計算自己的 transform tween。父用 `SPRING`、子沒寫 → 子吃 motion 預設 spring（stiffness 較低、感覺較軟）。視覺上是「父先到位、子還在路上」的拖尾感，看起來怪。

Transition 不像 CSS variable 那樣 cascade，是 prop 一份一份算。每個 motion 子都記得帶。

## Shrinkwrap：避免文字 re-flow 問題

文字置中（`text-align: center`）+ 寬容器時，`<p>` 自己佔滿寬度，文字浮在中間。窄容器時 `<p>` 還是滿寬，但文字 wrap 兩行。

Motion 看的是 `<p>` 的 bounding box（寬高），不知道字實際在哪。寬→窄轉換時 motion 把 `<p>` 從「寬橫條」scale 到「窄方塊」— 反向 transform 算的是 box 中心，但**字的 wrap 行數變了**，視覺上會看到字「邊跑邊重排」。

解法是**讓 `<p>` 自己 shrinkwrap 到字寬**，例如父用 `flex justify-center`，`<p>` 變 inline-block 寬度等於字寬。Bounding box 永遠等於文字實際 footprint，motion 計算才準確。

我這個 demo 父用 `flex items-start justify-center` + `<motion.p className="m-0">`，效果同。

## Reorder：array 順序變 → motion 自動補位

`layout` 不只用在「展開縮放」。任何讓 React 重新計算 layout 的東西都觸發 — 包括 array reorder。

```tsx
{items.map((item) => (
  <motion.div key={item.id} layout transition={SPRING}>
    {item.label}
  </motion.div>
))}
```

只要 key 穩定，React 知道是同一個物件「換位置」，motion 就量舊 y、量新 y、用 `translateY` 補幀。

## Gotchas

- **`layout` prop 只觸發在「同一個 motion 元件 render 之間 bounding box 改變」**。如果你 conditional 改成 `{cond && <motion.div layout/>}`，元件被 unmount → mount，沒有「之前的 box」可以比對 — 用 `AnimatePresence` 才能管 mount/unmount 動畫。
- **transition 不 inherit**，父子都要重複寫 `transition={SPRING}`。少寫一個就看到「父快子慢」的拖尾。我把 SPRING 抽成 module const + spread 進兩邊，避免不同步。
- **內層用 `layout="position"` 不是 `layout`**。後者會讓子也計算 size，等同沒抵消父的 scale。
- **文字置中要 shrinkwrap**。父用 flex justify-center 就夠，不要在 `<p>` 上設 `width: 100%` + `text-align: center`，那會讓 motion 看到「寬度變了的 box」反而觸發更多動畫。
- **`as const` 的 SPRING 物件**:`type: 'spring'` 必須是 literal type，否則 TS 推成 `string`、motion 的 transition prop 會拒絕。寫 `{...} as const` 一勞永逸。
- **HMR + layout animation 互動很糟**。修 code 時 motion 常認不出「之前的 box」造成怪 jump。Josh 的 playground 直接停 HMR 改 full reload — 這份 demo 用 Vite HMR 也會偶爾跳，存檔後手動重新整理最穩。
- **Container 必須是 positioning context**。我 demo 的外層 `<div className="relative ...">` 是必要的 — `wrapperMaximized` 用 `absolute inset-3`，`absolute` 會找最近的 positioned ancestor 對齊。漏 `relative` 會跑去對齊整個 viewport。
- **`motion.p` 預設 margin 不是 0**。`prose` style 下 `<p>` 有上下 margin，會讓 layout box 含進去。我加 `className="m-0"` 移除，否則文字位置會「跳」一個 margin 高度。
