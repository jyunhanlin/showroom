## TL;DR

- 核心招式：`stroke-dasharray: <pathLength>, 9999` 讓整條 path 只有開頭那段是實線，後面 9999px 全是 gap。
- **方法 A（過時）**：把 dasharray 從 `0, 9999` 動到 `fullLen, 9999` — dash 從 0 長到滿。
- **方法 B（推薦）**：dasharray 維持 `fullLen, 9999` 不變，把 `stroke-dashoffset` 從 `fullLen` 動到 `0` — 整段 dash 從 path 外滑進來。
- 為什麼 B 比較好：`stroke-linecap="round"` 時 dasharray=0 會留一個圓點殘影（round cap 把 0 寬 dash 撐成 circle），方法 A 動畫起點/終點看起來不乾淨；方法 B 整段 dash 滑出畫面 = 完全消失。
- `pathLength="100"` 設一個虛擬長度，CSS 端 dasharray / dashoffset 的數值都對 100 來算。免去 `getTotalLength()` JS 算路徑真實長度。
- 自我繪製是 module 7 自身完整的 demo trick，但跟 SVG path / stroke / dash 那一連串 building block 全部串起來的 capstone。

## 兩個方法的差別

```css
/* 方法 A：dash 變長 */
.scribble {
  stroke-dasharray: 0, 1000;
  transition: stroke-dasharray 1000ms;
}
button:hover .scribble {
  stroke-dasharray: 365, 1000;
}

/* 方法 B：dash 滑進來 */
.scribble {
  stroke-dasharray: 365, 1000;
  stroke-dashoffset: 365px;
  transition: stroke-dashoffset 1000ms;
}
button:hover .scribble {
  stroke-dashoffset: 0px;
}
```

兩者都從「看不到」動到「完全顯示」，但實作差別：

- 方法 A：dash 長度從 0 變大到 fullLen
- 方法 B：dash 長度恆定 fullLen，整個 pattern 在 path 上往前滑

`stroke-linecap: round` 時：

- 方法 A 起始狀態 dasharray=0 會被 round cap 撐成圓點 → 動畫**起點殘影**一個圓
- 方法 B 起始狀態 dasharray=fullLen 但 dashoffset=fullLen → 整段 dash 推出 path 範圍 → 完全看不見，**乾淨**

## pathLength 是 quality-of-life

```svg
<path pathLength="100" d="M 5,25 C ... 154,6" />
```

```css
.scribble {
  stroke-dasharray: 100, 1000;
  stroke-dashoffset: 100px;
  transition: stroke-dashoffset 1000ms;
}
.scribble:hover {
  stroke-dashoffset: 0px;
}
```

`pathLength` 屬性宣告「**把這條 path 當作 100 px 長**」（即使真實長度是 365.6px）。瀏覽器內部 normalize：CSS 寫的 100 對應到真實 365.6，視覺上整段 dash 滑入完整。

好處：

- 不需要 `getTotalLength()` JS API 算精確長度
- 多條長度不同的 path 可以共用同一套 CSS（都假設 100）
- 改 path data 不影響 CSS（真實長度變了但虛擬長度還是 100）

代價：「1px」在不同 path 上代表不同實際 px。dash 跟 gap 的視覺長度也跟著縮放，要做「真實 1px」效果就不能用 pathLength。

## getTotalLength() — 動態 path 的補救

```js
const elem = document.querySelector('path');
const pathLength = elem.getTotalLength();
console.log(pathLength); // 365.54931640625

elem.style.strokeDasharray = `${pathLength}, 1000`;
elem.style.strokeDashoffset = `${pathLength}px`;
```

要 path data 從 server fetch 進來、或 React state 動態算的場合，`pathLength` attribute 可能無法靜態指定（不知道何時更新）。改用 `getTotalLength()` 在 mount 後計算，回填 dasharray 跟 dashoffset。

效能：`getTotalLength()` 不便宜（瀏覽器要走完整條 path 累積長度）。**只在 mount 時呼叫一次**，存進 ref/state。每幀呼叫會卡。

## px 後綴的 quirk

```css
.scribble {
  stroke-dasharray: 100, 1000;     /* unitless 在這裡是 px */
  stroke-dashoffset: 100px;        /* 必須加 px */
}
```

`stroke-dasharray` unitless 數字 spec 上是 user units (= px)，多數瀏覽器接受。

但 `stroke-dashoffset` 在 Firefox 的 CSS context 必須有 `px` suffix，沒寫的話 Firefox silently ignore。

**安全策略：CSS 裡所有 SVG 數值都加 `px`**。Josh 自己也說他常忘記，但記得加比較不會踩到。

## Gotchas

- **動畫結束「圓點消失」現象**：方法 B 倒回（hover-out）時整段 dash 滑出 path → 最後變成 0 寬度的 dash → round cap 撐成圓點 → 視覺上「啪」消失。要解可以用 mask 或繼續滑超出畫面。Josh 後面 mask 那課會講。
- **效能差距 ~17%（從 Josh 自己的 benchmark）**：方法 A 寬度持續變的 dash 比方法 B 滑動的 dash 多 ~20% paint cost。但兩者在 16.6ms budget 內都遠遠夠用，2.2% vs 1.8%，肉眼看不出。
- **JSX 寫法**：`pathLength` 是 SVG attribute 直接寫 `<path pathLength="100" />`。Tailwind v4 的 `[stroke-dashoffset:0px]` arbitrary class 可以套 group-hover 觸發。
- **不要用 stroke-dashoffset 做大量 path 同時動**。本身一張單路徑很便宜，但 100+ 條 path 同時跑 dashoffset 動畫會在低階機器掉幀。考慮拆成 IntersectionObserver 觸發或限制動畫條數。
